简版的koa实现

Posted on 2020-05-21 Views: Valine:

问题描述

初学者刚接触 nodejs 的时候应该大部分都是使用 koa 去应对日常开发练习,但是,就有一个问题,如果过个半个月一个月不去整,我就会忘记怎么写,连入口文件都忘记了,思来想去,还是对koa这个东西不了解,停留在看文档写(抄)代码的阶段,所以,参照着koa的源码,动手写了一个简化的koa,源码在这传送门

实现过程

我这里实现了koa基础功能,static静态文件服务,以及router插件

  • 先初始化项目npm init -y

  • 新建Vkoa文件夹,index.js以及public文件夹

  • index.js 入口文件,和使用koa无异

  • Vkoa.js 文件,即 koa 的主体文件
    这个koa的核心文件,我们平常使用的use,listen,use这个函数将使用app.use()注册的函数保存在fnList中,listen这个函数启动了一个http服务器,同时调用其他函数处理这个请求,compose是聚合函数,用于将我们注册的函数聚合,createContext用于将request和response绑定到ctx上面,同时提供一些常用属性如url,method的快捷读取.
    这里的compose和createContext这两个方法特别是compose方法应该是比较迷的地方了.

compose函数,这个函数也是我们能实现洋葱模型的关键,洋葱模型可以理解为一系列函数层层嵌套执行,像我们写的:

这段代码的执行结果为:1,2,3,4,5,6
从外到内,再从内到外,每个 next 会等待下面的函数执行完后再执行下面的代码,这只是一个简单的例子,但是为什么调用next函数会往下执行呢?这就是compose这个函数的作用了.
我们可以看到,每调用一次app.use(),use函数会将传入的函数保存到一个数组中,然后让compose调用,compose返回一个匿名函数,这个匿名函数调用其自生声明的一个函数toNext,toNext函数以传入的下角标去获取对应的函数,即我们前面app.use()传入的函数,函数存在,就返回一个Promise,resolve的结果就是用下角标获取的函数的执行结果,里面的这个函数以ctx和一个函数,这个函数执行下一个toNext并且将下角标加一,这就是为什么我们在执行next后会执行下面的函数的原因,注意这里一定要返回一个Promise,后面会将原因.

createContext函数,这里的这个Object.create()方法我也是头一次将,可以理解为一个继承操作,新建一个对象并且继承子传入的参数,可以看到,我们创建一个ctx对象并且继承自context,context具体见下面,同时在其上新建request以及response属性,同时将真正的request和response绑定到ctx.req以及ctx.res上面.最后返回ctx,源码封装了更多快捷操作在ctx上面.

  • request.js, response.js 以及 contenxt.js
    这三个模块主要用于定义访问器实现属性的快捷操作,对访问器不懂得小伙伴,就记得这是一个属性拦截器就好了.
  • router 的实现
    看,我们平时执行的app.use(router.routes)就是这么回事,返回了一个函数,我们所声明的router.get(),router.post()都被存到了一个数组里面,每次有新的请求到达时,就会去遍历这张表,找到mehtod和url符合的项,然后执行它.
  • static 的实现
    static 的实现比较简单,根据指定的url和自定义的静态文件路径去访问文件,注意一定到返回一个异步函数,因为文件读取时一个异步操作,你不使用 await,会直接过!

总结

这个算是比较简单的版本,但是我也遇到了一些问题,就是前面那个compose函数一定要返回一个Promise的地方,我可能对nodejs的异步操作理解的太烂了,如果不返回一个promise,你在listen这个函数里面执行compose的返回结果的时候,使用await是没用的,那个地方如果不使用await,只要你使用app.use()注入的函数是一个异步函数,里面的任何异步操作,会被直接跳过,在里面使用await无效,如果你想在一个get请求里面读取文件,然后将文件读取结果返回,他会跳过这一步,我就郁闷,我注册的时候不声明异步函数,他也会直接跳过,en,又遇到问题了,等我想明白再更新!我觉得我应该不会忘记koa入口文件怎么写了!
还有那个static和router之间不是洋葱模型,即如果你把app.use(staitc())写在app.use(router.routes())前面的时候,不会执行到路由里面去,反之亦然!
